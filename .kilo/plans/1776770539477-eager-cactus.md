# 1. Frontend Differences

## 1.1 New Component: ChatRegisterBar (AI Chat Registration)

The reference project introduces a **634-line AI conversational registration component** (`ChatRegisterBar.tsx`) that does not exist in our project. This is the single biggest difference.

- **AI chat interface**: Uses `supabase.functions.invoke("chat-register")` to call GPT-5-mini via Lovable AI Gateway
- **Progress tracking**: Tracks 7 required fields (category, fullname, country, city, field, email, phone) with a visual progress bar
- **Multi-file upload**: Up to 5 files (PDF, DOC, DOCX, JPG, PNG, WEBP), 10 MB each, uploaded to `submission-documents` Supabase Storage bucket
- **AI Match Preview**: After submission, calls `find-matches` edge function and displays match cards inline with score, name, city, field, and AI-generated reason
- **Classic form fallback**: "Klasik Form" button opens `RegisterInterestForm` dialog
- **City pre-fill**: Listens for `corteqs:select-city` custom DOM event to pre-fill city field

**Our project**: No chat registration component. Only traditional dialog forms (`RegisterInterestForm`, `BackerForm`).

### find-matches Edge Function
- **Step 1**: Tokenizes query (TR+EN stopwords), calculates keyword overlap against top 200 recent submissions
- **Step 2**: Top 20 keyword candidates ‚Üí Gemini 2.5 Flash for semantic ranking
- **Output**: Top 5 matches with score (0-100) and Turkish reason
- **Persistence**: Optionally writes to `matches` table via `persist=true` param
- **Environment**: Requires `LOVABLE_API_KEY`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`

# 3. Gap & Opportunity Analysis

## Must Adopt

1. **ChatRegisterBar component** ‚Äî This is the reference project's core differentiator. AI chat registration dramatically reduces friction for users compared to a long form. High UX impact.

2. **chat-register edge function** ‚Äî Required backend for the chat component. Uses Lovable AI Gateway with GPT-5-mini. Must be deployed to our Supabase project.

3. **find-matches edge function** ‚Äî AI-powered offer/need matching is the platform's core value proposition. Must be deployed alongside chat-register.

4. **`offers_needs` column** ‚Äî Required for the AI matching system to work. The matching algorithm relies on this field.

5. **`matches` table + migrations** ‚Äî Required for persisting AI-generated matches and the notification pipeline.

6. **File upload system** (`document_url`, `document_name`, `documents` JSONB, `submission-documents` storage bucket) ‚Äî Users uploading CVs and documents is a core feature for a professional networking platform.

7. **Expanded cities** (9 cities + global marquee) ‚Äî Our 6-city Germany-focused list is too narrow for a "global diaspora" platform. The reference's international scope is correct.

8. **Additional categories** (`vakif`, `radyo-tv`) ‚Äî These are real diaspora segments. Missing them loses potential registrations.

## Nice to Have

1. **City-specific CTA buttons** (AI Chat + Form per city card) ‚Äî Good UX improvement but can be added incrementally.

2. **Cross-component event bus** (`corteqs:select-city`) ‚Äî Enables city pre-filling in both chat and form. Clean pattern.

3. **Category contextual hints** (e.g., danisman shows specialist info) ‚Äî Improves form completion rates.

4. **AI match hint text** in forms ("ü§ñ Detaylƒ± veri AI e≈üle≈üme kalitesini artƒ±racaktƒ±r") ‚Äî Encourages users to provide richer data.

5. **CHANGELOG.md** ‚Äî Good development practice for tracking session-by-session changes.

## Not Applicable

1. **Removing admin review workflow** ‚Äî Our `admin_users` table, status workflow, and notes system are **superior** to the reference. The reference has no admin check at all ‚Äî any authenticated user sees all submissions. We should NOT adopt this regression.

2. **Removing email notifications** ‚Äî Our `send-submission-email` edge function provides admin notification and optional user confirmation. The reference dropped this. We should KEEP our email system.

3. **Removing AboutPage** (`/hakkimizda`) ‚Äî Our separate about page with SEO is better than having none.

4. **Removing FormPage** (`/form`) ‚Äî Standalone form page is useful for direct linking.

5. **Removing FAQSection** ‚Äî 12-item FAQ is valuable for SEO and user education.

6. **Removing Docker/deployment setup** ‚Äî Our Dockerfile + nginx.conf is production-ready. Reference has none.

7. **Removing SEO infrastructure** ‚Äî Our structured data, analytics, sitemap, robots.txt, llms.txt are far more comprehensive. Reference has basic SEO only.

8. **Removing centralized helpers** (`lib/submissions.ts`, `lib/admin.ts`, `lib/mail.ts`) ‚Äî Our code organization is cleaner. Reference has inline data construction.

9. **Removing unit tests** (`submissions.test.ts`) ‚Äî Reference has only a placeholder test.

10. **Removing BloggerSection from Index** ‚Äî The reference excluded it from the page layout, but it's still a valid recruitment section.
